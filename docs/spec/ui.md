# UI / UX

本製品の主UIは対話、ファイル、コマンド案内、作業報告である。
Chatworkのルーム設定と、Google ChatのOAuth・スペース設定は、loopbackで開く1つの共通ローカル設定wizard骨格を使う。
対象は技術に多少関心がある非エンジニアであり、**専門家同士の正確さと頭に入りやすさの中間**を目指す。

## 体験の原則

1. **最初に「何が起きているか」「何をすればよいか」を示す**。背景説明は必要な分だけ後に置く。
2. **一般技術用語はそのまま使う**。ディレクトリ、ファイル、インストール、コミット、プラグイン、コネクタ、TODO等を幼稚な比喩で置き換えない。
3. **馴染みの薄い語だけ初出で短く補足する**。OAuth、MCP、frontmatter等。基本語を毎回説明しない。
4. **報告は既定3行**。「やったこと／結果／次に何が起きるか」。設定が「くわしく」の場合だけ、3行＋補足1つへ拡張できる。
5. **複数手順は進行を見せる**。計画→実装→検証の正式名称を保ちつつ、必要なら計画→道具→確認→結果を併記する。
6. **英語エラーを翻訳する**。原文は検索用に残してよいが、何が起き、どう直すかを先に日本語で示す。
7. **不安を先回りする**。時間、確認、失敗しうる箇所を着手前に一言で伝える。

## やさしさの定義

やさしさは、言葉遣い、報告、進行の見せ方、次の一手の提案に現れる。
規律、3 Agent分離、根拠、記憶保護、封じ込め、評価閾値、回帰ゼロ許容を省くことはやさしさではない。

### 先回り提案

- 報告3行目を、可能なときは「次の一手の提案」にする。
- 提案は1つまで。根拠を一言添え、実行するかはユーザーに委ねる。
- ユーザーの明示指示なしに、提案した作業へ勝手に着手しない。
- 適切な提案が無いときは無理に作らず、次に起きることを述べる。

## オンボーディング

初回は5問以内。

1. 呼び方
2. 主に使うサービス
3. 秘書に任せたいこと
4. お仕事・役割（自由回答＋例）
5. 説明の詳しさ（3択1タップ）

口調は初回に聞かない。丁寧で堅すぎない標準設定で開始し、完了報告で「いつでも『設定変えたい』で変えられます」と伝える。
既存 `secretary/` がある場合は通常オンボーディングへ進まず、バックアップ提案と明示確認を行う。

秘書設定後は、同じ導線の中でprivate GitHub repoの名前と保存内容を確認し、repo作成、初期commit、初回pushまで完了する。
既存remoteを検出した場合は別repoを作らず、現在のrepoを使うか確認する。
完了画面はprivate状態、remote、初回push結果と、Chatworkまたは高度なGoogle Chat接続へ進む次の1手を示す。

## Chatwork／Google Chat共通の文章設計

既存の接続・選択・保存機能は変えず、主導線の文章を **More Simple, Less is More** に揃える。
利用者が最初に読むのは内部の仕組みではなく、「今すること」「次に起きること」「安全上の判断」である。

### Primary pathとtechnical detail

- 画面冒頭は `今すること: <1文>` の意味を持つ短い文から始める。見出しと重複する場合はラベルの機械的一致を求めないが、最初の本文だけで行動が分からなければならない。
- 1画面で求める判断は1つ、1段落は1要点。準備、権限、安全同意、実行結果を同じ段落へ詰め込まない。
- `wizard`、`workflow`、`commit・push`、`Repository Secret`、`loopback`、`runtime`、`scope`、`token`、`OAuth client JSON`、Sprint番号等は、主説明より先に目的を日常語で示す。判断に不要なら「詳しい説明」または「管理者向け」へ退避する。
- API、OAuth、Google Cloud、GitHub Actions等の正式名称は、設定や管理者依頼に必要な箇所で残す。初出は「Googleアカウントで接続を許可する仕組み（OAuth）」のように短く役割を添える。
- technical detailは閉じた状態を既定にし、開かなくても通常導線を完了できる。管理者に渡す識別子・正式な設定名・英語エラー・診断情報は、必要になった画面だけに置く。

### 画面ごとの最大情報量

| 画面の役割 | 主表示に置ける情報 | CTA上限 |
|---|---|---:|
| 開始・状態確認 | 見出し1つ、今すること1文、現在状態1行 | 2 |
| サービス固有の準備 | 今すること1文、目的1段落、完了条件1行。正式手順はdetailsへ | 2 |
| 接続確認 | 今すること1文、安全な扱い1段落、確認結果1行 | 2 |
| 対象選択 | 今すること1文、選択数1行、検索・一覧。安全説明は確認画面へ | 2 |
| 取得間隔 | 今すること1文、3時間推奨の理由1行、選択肢 | 2 |
| 保存前の確認・同意 | 今すること1文、安全上の意味を最大5項目、選択内容の要約 | 2 |
| 処理中 | 現在行っていること1行、待つ理由または進捗1行 | 1 |
| 0件・手動のみ | 今後止まる処理1文、履歴保持1文、選べる次の行動1文 | 2 |
| 失敗 | 何が起きたか1段落、次にすること1段落 | 2 |
| 完了 | 確認できた結果1文、次の一手1文 | 1 |

一覧そのもの、選択肢、法的・安全上欠かせない短い項目は上の文数へ機械的に数えない。ただし、同じ意味の重複や、判断に不要な補足を主表示へ足す理由にはしない。

### 安全情報と状態copy

- 確認画面では「読む対象」「保存先」「共同編集者にも見えること」「自動取得・自動保存」「履歴を自動削除しないこと」を、1項目1要点で示す。
- Google Chatの `read-only` は主説明で「メッセージを読むだけで、投稿や編集はしません」と示し、正式なscope名は管理者向け詳細へ置く。通常スペースだけが対象で、DM／グループDMを含めないことは選択画面と確認画面の両方で分かるようにする。
- 選択0件は「選択を0件にすると、今後の取得は止まります。これまでの履歴は削除しません。」の意味を持つ自然な日本語にする。
- 手動のみは「自動取得を止めます。必要なときだけ取得でき、これまでの履歴は残ります。」の意味を持つ自然な日本語にする。
- エラーは「接続を確認できませんでした。接続情報を見直して、もう一度お試しください。」のように、状態名やコードより先に原因の範囲と次の行動を示す。原因を特定できないときは推測で断定しない。

### CTAと言語品質

- CTAは「接続を確認する」「保存内容を確認する」「この設定で始める」のように、短い動詞句で次の結果を示す。
- 不自然な直訳、主語が無く対象を判断できない文、英日混在、同じ意味の二重表現をcopy inventoryの段階で洗い出す。
- サービス名は全画面で明示し、Chatwork／Google Chatを取り違えない。色だけを識別手段にしない。

## Chatwork設定wizard

共通wizardの全画面で、可視見出しとスクリーンリーダー向けのaccessible nameに「Chatworkの設定」を明示する。
Google Chatとstep骨格・responsive・accessibilityを共有しても、対象サービスを文言とprimary CTA色の両方で取り違えないようにする。

### 体験フロー

1. `/chatwork` は最初に「未接続／ルーム選択待ち／同期済み／要確認」の状態と、次の行動を1つ示す。
2. 未接続では「ChatworkでAPI Tokenを取得する」から始め、公式Tokenページと発行ヘルプを新しいタブで開けるようにする。Tokenはwizard・会話・repoへ入力させない。
3. パーソナルプランを除き組織管理者へのAPI利用申請が必要である。Tokenページを利用できない場合は、「組織管理者へAPI利用申請→承認後に戻る」を同じstepで示す。実際にAPIを使うアカウントで申請すること、管理者が承認または却下することを説明し、承認前は先へ進めない。戻った時に設定途中の選択を失わない。
4. 現在のrepoのowner／nameからSecret追加URLを組み立て、「GitHub上の安全な保管場所を開く」で新しいタブを開く。名前 `CHATWORK_API_TOKEN` で登録し、登録できたことを利用者が確認するまでルーム一覧取得へ進まない。
5. 登録確認後、決めた間隔で自動取得を動かすGitHubの仕組みであるGitHub Actionsが参加中のルーム一覧を同じ非公開のGitHubリポジトリへ反映し、wizardをローカルで開く。0ルーム、取得失敗、認証失敗は別メッセージにする。
6. ルーム選択ではルーム名を主表示、ルームIDを補助表示し、検索と全解除を提供する。初期状態で全ルームを選ばない。
7. 自動取得の間隔では「30分ごと／1時間ごと／3時間ごと（おすすめ・初期値）／6時間ごと／12時間ごと／手動のみ」と30日換算の概算実行回数1,440／720／240／120／60／0回を並べる。
8. 確認stepで、対象ルーム、自動取得の間隔、保存先が同じ非公開のGitHubリポジトリであること、共同編集者が本文を読めること、最新100件制約、自動保存への同意を1画面にまとめる。「取得結果をこのリポジトリへ自動保存します（Gitのcommit・push）」と目的を先に示す。
9. 確定後に初回取得を実行し、0件でも成功として「今後の最新メッセージの取り込み（同期）から蓄積される」と伝える。成功ルーム／失敗ルーム／取得件数を区別する。
10. 既存設定の変更後は、現在の選択ルーム、現在の自動取得の間隔、自動実行の有効／無効を結果stepに表示する。「初回設定の結果」と変更前のルーム・取得件数を再表示しない。

各stepは1つのprimary messageに絞り、CTAはprimaryとsecondaryの最大2つ。
戻る操作で選択値を失わず、確定前のキャンセルはrepo・workflow・履歴を変更しない。

### 表示用語と情報量

- ユーザー向けの `room` は原則「ルーム」、識別子が必要な箇所は「ルームID」とし、初出で「ルームを識別する番号」と短く補足する。
- `頻度` は「自動取得の間隔」、`runs` は「実行回数」、`schedule` は「自動実行」、`workflow` は「自動取得処理（GitHub Actions）」、`private repo` は「非公開のGitHubリポジトリ」とする。
- `Repository Secret` は初出で「GitHub上の安全な保管場所（Repository Secret）」とする。API Token、GitHub Actions、commit、pushなど検索・設定に必要な正式名称は残す。
- `同期` の初出は「最新メッセージの取り込み（同期）」とする。英語だけの見出し、状態、略語を残さず、操作結果と次の行動が日本語で分かるaccessible nameを付ける。
- 1 step 1 message、CTA最大2を守る。選択に必要な説明は本文、料金や処理時間の補足は「料金と実行時間について」のdetailsまたは短いhelpへ分ける。
- 料金補足は、実行回数と処理時間が別であること、GitHub Freeの非公開リポジトリでは2026年7月時点で月2,000分が目安であること、プランと1回の処理時間で変わること、料金・枠が変更されうることを短く示し、GitHub公式billingページへ案内する。
- 外部リンクは新しいタブで開き、「ChatworkでAPI Tokenを取得する」「API Tokenの発行方法を見る」「組織契約のAPI利用申請を見る」「GitHub Actionsの料金と利用枠を見る」のように行き先と目的を日本語で示す。

### デザイン言語

- Pure White `#FFFFFF` とLight Ash `#F4F4F4`を面に使い、見出しはCarbon Dark `#171A20`、本文はGraphite `#393C41`、補助はPewter `#5C5E62`とする。
- Chatworkのprimary CTA背景は `#F03747`、Google Chatのprimary CTA背景は `#11BB62` に固定し、両方の前景を `#000000` にする。背景色を薄めず文字・アイコンとのcontrast ratio 4.5:1以上を満たし、従来の青色primary CTAを残さない。
- サービス別色はprimary CTAだけに使い、状態ラベルや装飾へ広げない。選択・成功・失敗は色だけで伝えない。
- gradient、shadow、pattern、装飾borderを使わず、余白で階層を作る。入力やfocusに必要な最小borderはCloud Gray／Pale Silverを使う。
- system sansを使い、400と500の2 weight、本文・label・buttonは14px中心、headlineはdesktop最大40px、mobile目安28pxとする。
- interactive elementは4px radius。8px spacing systemを使い、desktopは中央寄せで広い余白、1画面1メッセージの静かな構成にする。
- transitionはcolor／background／borderを0.33秒。hoverでscale・translateを使わない。`prefers-reduced-motion`を尊重する。
- Teslaの商標、wordmark、車写真、Universal Sans等のライセンス不明フォントを使わない。添付資料の視覚原則を応用し、Teslaサイトの複製にしない。

### Responsive・accessibility

- 768px未満は1 column、CTA縦積み、room項目をfull widthにする。desktopは選択一覧と説明を読みやすい最大幅に収める。
- keyboardだけでstep移動、room選択、頻度選択、確定、キャンセルができる。focusを明確に表示する。
- checkbox、radio、入力、エラーに可視labelと関連付けを持たせる。色だけで選択・成功・失敗を伝えない。
- touch targetは44px相当を確保し、200% zoomでも情報や操作を失わない。

## `/chatwork search` の対話導線

- まずpull中であることを短く示し、保存済み履歴を検索する。見つかった場合はroom名、日付、該当箇所を返す。
- 見つからない場合は、AskUserQuestionまたはCodexのstructured input等の構造化質問で「同期して再検索（推奨）」「同期しない」「対象roomを見直す」を提示する。
- 同期承認後は「workflow開始→待機→成功確認→pull→再検索」の現在地を示す。失敗やtimeoutを黙って再試行しない。
- 再検索でも見つからない場合は、導入前／100件制約／未選択room／キーワード／編集・削除／同期失敗のどれが残るかを示し、存在しないと断定しない。

## Google Chat設定wizard（少し高度な設定）

Chatworkと同じ共通wizard骨格を使い、全画面の可視見出しとaccessible nameに「Google Chatの設定」を明示する。

### 位置づけ

- READMEでは基本の導入・Chatwork説明より後ろに「Google Chatをつなぐ（少し高度な設定）」として置く。
- 冒頭で「Google Workspace管理者、またはGoogle Cloudプロジェクトを作成できる方が必要」と示し、通常利用者が自分だけで進められない可能性を隠さない。
- OAuthは初出で「Googleのパスワードを渡さず、許可した範囲だけ読み取る認証」と短く補足する。Restricted scopeは「読み取り専用でもGoogle上では重要データへの権限」と説明する。
- 管理者に依頼する人向けに、Cloud project所有組織、`Internal` Audience、Chat API／People API、Desktop app clientという依頼事項をコピーできる短いチェックリストにする。

### 体験フロー

1. `/google-chat` は「未準備／管理者作業待ち／Google認証待ち／スペース選択待ち／同期済み／再認証が必要」の現在状態と、次の行動を1つ示す。
2. 未準備では、各社所有Google Cloudプロジェクトを作り、Google Chat APIとPeople APIを有効にし、OAuth Audienceを `Internal`、Clientを `Desktop app` にする手順を画面順に案内する。
3. OAuth client JSONをダウンロードしたら、wizardでローカルファイルとして選ぶ。内容やclient secretを画面、会話、ログへ表示せず、外部へuploadしないことを明示する。client IDは識別子として扱い、管理者チェックリストへ必要時だけ一時表示できるが、再読込後へ残さない。
4. 「Googleで認証する」で既定ブラウザを開き、PKCEとstate検証を伴うloopbackで同意結果を受け取る。要求するread-only scopeと、何を保存するかを認証前に日本語で示す。拒否・callback不一致・state不一致・管理者ブロックを別メッセージにする。認可URLとcallback URLは画面・ログ・証跡へ表示せず、認可コードは受領後すぐ交換する。
5. 認証成功後、資格情報を現在のprivate repoのRepository Secretへ直接登録する。値のコピー＆ペーストや会話への貼付けを通常手順にせず、登録完了だけを表示する。
6. 通常スペース一覧では名前を主表示、space IDを補助表示し、検索、複数選択、全解除を提供する。初期状態は0件。DM／グループDMは初版対象外として一覧に出さない。
7. 初回取得前の確認画面で、対象スペース、取得可能な履歴、スレッド、発言者、添付メタデータ、添付本文を取らないこと、private repo共同編集者が本文を読めることを示す。「取得結果をこのリポジトリへ保存します（Gitのcommit・push）」を独立項目にし、明示同意後だけ進む。
8. 自動取得の間隔は「1時間／3時間（おすすめ・初期値）／6時間／12時間／手動のみ」。3時間はChatworkと共通の推奨であることを、利用者向けには「負担と新しさのバランス」として説明する。
9. OAuth前のキャンセルはrepo、Secret、設定、workflow、履歴を変更しない。OAuth同意後にキャンセルする場合は、作成済みSecretの削除とGoogle側のOAuth grant／token revokeを案内し、アプリ権限ページで取消できるようにして曖昧な接続状態を残さない。
10. 初回取得は同じwizardセッション内でメモリ上のtokenだけを使ってローカル実行し、進捗、成功スペース、失敗スペース、日付ファイル数、メッセージ件数を示す。終了時にtokenを破棄し、0件も正常として、取得可能範囲と組織の保持設定の影響を説明する。
11. 発言者名はPeople APIで補完できない場合があり、そのときは安定した代替表示になることを故障ではなく仕様として示す。

### エラーと再認証

- `redirect_uri_mismatch` は「Cloud ConsoleのOAuth client種別またはcallback設定が一致しない」と日本語で示し、検索用に正式なエラー名を残す。
- `access_denied` は利用者の拒否、Workspace管理者のブロック、Audience不一致を分けて確認する。
- refresh token失効、scope不足、API無効、rate limit、network失敗を同じ「接続できませんでした」にまとめない。
- 再認証では既存スペース選択と履歴を保持し、新しいrefresh token登録が成功した後だけ接続済みに戻す。

### デザイン・accessibility

- Chatwork wizardと同じ共通骨格、4px radius、8px spacing、CTA最大2、responsive、keyboard、focus、label、contrastの基準を使う。Google Chatのprimary CTA背景は `#11BB62`、前景は `#000000` に固定する。
- Cloud Consoleの手順は長い1ページにせず、「管理者準備」「OAuth client」「Google認証」「スペース選択」「取得設定」に分け、現在地と戻り先を表示する。
- 外部リンクは「Google Cloudでプロジェクトを開く」「Google Chat APIを有効にする」「OAuth同意画面を設定する」のように、行き先と目的が分かるラベルにする。

## `/google-chat search` の対話導線

- まずpull中であることを短く示し、保存済み履歴を検索する。見つかった場合はスペース名、日付、該当箇所を返す。
- 見つからない場合は構造化質問で「取得して再検索（推奨）」「取得しない」「対象スペースを見直す」を提示する。
- 取得承認後は「workflow開始→待機→成功確認→pull→再検索」の現在地を示す。再認証が必要ならworkflowを繰り返さず、先に認証導線へ戻す。
- 再検索でも見つからない場合は、未選択スペース、組織の保持設定、API取得範囲、キーワード差、編集・削除、同期失敗を示し、存在しないと断定しない。

## 節目プロトコル

### 決定

- 「〜にしよう」「じゃあそれで」「それで決定」等を決定の合図として扱う。
- 既定は、原文を勝手に膨らませず「この内容を決定として残しますね: …」の1行確認を行う。
- `preferences.md` が「まとめて」の場合は、会話の締めで候補をまとめて確認する。
- 締めで当日の `decided` が0件なら、会話を読み返して拾い漏れが無いか確認する。

### 相談の文脈

- 結論がなくても案件の背景として価値がある相談は、一区切りで「要点を案件メモに残しますね」と1行確認する。
- 了承後に要点だけをtopicsへ残す。逐語ログは保存しない。

### 活動

- 成果物保存やTODO完了等、定義済みシームが成功した事実は確認を増やさずjournalへ追記する。
- 自動追記をしたことは通常の節目報告に含め、対話を細かな確認で止めない。

## プロジェクトの対話導線

### 候補の提案

- 複数行動・複数セッションを含む候補シグナルが揃ったときだけ、理由を1〜2点に絞って提案する。
- 既定文は「この内容は今後も続きそうです。プロジェクトとしてまとめますか？」とし、構造化質問で「まとめる／今回はまとめない」を選べるようにする。
- 確認前にファイルを作らず、拒否・キャンセル時は作成、journal、commitを0件にする。単発作業へ何度も提案しない。

### 作成と昇格

- 一般PJの承認後は、プロジェクト名、ゴール、現在状況、最初の次の入口を短く確認し、`PROJECT.md` 1枚のライト運用を作る。既存情報がある場合は空テンプレではなく実内容を入れる。
- 完了報告は、作成場所、現在状況、次の入口を既定3行で示す。内部のテンプレ説明を長く列挙しない。
- 昇格トリガー到達時は「情報が増え、状態・判断・事実を分けた方が探しやすい」と理由を示し、「フル運用へ整理する／今はライトのまま」を確認する。了承前にファイル移動・分割を行わない。
- 開発依頼は一般PJへ吸収せず、既存の `build` 導線を維持する。別repoを正本にする場合は、作成・接続・公開範囲と、workspace側には参照だけを置くことを先に確認する。

### 完了と再開

- 「このプロジェクトは完了」「終了にしたい」等では、対象PJと完了日、結果、残件の有無を短く示し、「完了扱いにする／まだ進行中」を確認する。確認前にstatusを変えない。
- 完了後は「完了扱いにしたこと／結果と残件を記録したこと／検索や明示参照は引き続きできること」を既定3行で伝える。自動archive・移動・削除は行わない。
- 完了済みPJに新しい作業が出た場合は、通常の候補提案を重ねず「このプロジェクトを再開しますか？」と確認する。了承後だけactiveへ戻し、再開日と理由を示す。
- 再開後も過去の完了記録を残し、拒否・キャンセル時はPROJECT、journal、commitを変更しない。

## settings 体験

- 「設定変えたい」「もっとフランクに」「専門用語そのままで」「呼び方を変えて」から入れる。
- 変更前に、その設定なら実際の報告がどうなるか短い例文を見せる。
- ユーザー確認後に反映し、「こう覚えました」と変更項目を宣言する。
- 「その言い方いいね」等を秘書のメモへ自発反映するときも、1行確認を挟む。
- 役割は質問で集めて終わりにせず、提案・例示・用語補足の題材へ反映する。

## 既定値と個人差

| 項目 | 既定 | 明示設定時 |
|---|---|---|
| 口調 | 丁寧・堅すぎない | フランク／きっちり敬語 |
| 専門用語 | ふつう | ことば添え／そのままOK |
| 報告 | みじかく（3行） | くわしく（3行＋補足1つ） |
| 決定確認 | 都度1行 | 会話の締めでまとめて |

設定が無い、空、壊れている場合は既定へ戻る。ユーザーの職業や反応から無断でcategorical設定を変えない。

## timeline の対話導線

- 「今日やったこと」「先週なにしてた」→ journalの期間表示。
- 「いつ決めた」「7月に決まったこと」→ decisions表示。固有名詞があればキーワード検索を併用する。
- 結果は日付、種類、本文が追えるMarkdownで返す。要約して原文の意味を変えない。
- 「保存して」で初めて成果物化する。

## build の参照導線

- 開発依頼時は `yasashii-harness` の存在を確認する。
- 未導入なら、何を入れるかを先に説明して3コマンドを案内する。導入済みなら正式な Planner → Generator → Evaluator の現在地を示す。
- Plannerを「段取り係」等と言い換えて隠さず、正式名称と短い役割説明を併記する。
- 別repo開発PJでは、workspace側に正本を複製せず、正本repoへの参照ポインタと概要スナップショットだけを持つことを明示する。

## 更新の対話導線

### 読み取り専用の診断

- 「最新版にして」「更新ある？」では、最初に現在版、最新版、主な変更、影響する設定・ファイル、必要な操作、カスタマイズ衝突可能性を表示する。
- 最新版を確認できないときは、現在版だけを示して「最新版は確認できませんでした」と説明する。古い情報から推測しない。
- 変更点は内部ファイル名の列挙だけにせず、「誰に関係するか」「今の使い方が変わるか」「利用者がすること」の順で短く示す。
- 自動更新は有効化できること、利点、注意点、利用者自身が行う設定手順を案内するだけとし、その場で設定を変更しない。
- 診断の末尾で「今回は確認だけ／実更新へ進む」を選べるようにする。無応答、拒否、キャンセルは確認だけとして終える。

### 明示確認後の実更新

- 実更新前に、更新対象、カスタマイズ判定、保護commitに含む範囲、pushしないこと、失敗時の戻し方をまとめて確認する。
- `customized` または `unknown-baseline` のファイルは1件ずつ「現状を残す（既定）／新版へ置き換える／差分を見る／中止」を選べるようにする。まとめて上書きを既定にしない。
- migrationはdry-runの追加・変更・維持対象を平易に示し、もう一度確認してから実行する。
- plugin更新後にreload／restartが必要なら、理由と再開方法を示す。再開後はversion、migration、主要導線を検証してから成功を報告する。
- 失敗時はpluginとworkspaceのどちらが変わったかを分け、直前commitと更新前versionを使うrollback手順を示す。pushは別操作として確認なしに行わない。

## 非機能要件

- macOS / Windowsで、ファイル構造と案内が破綻しない。
- Chatwork／Google Chat wizardは主要desktop browserで動き、外部interfaceへbindしない。OAuth callbackもloopback以外へ公開しない。
- `CC_SECRETARY_NOW` により日付依存の体験を固定して検証できる。
- 中断後は `_resume.md`、翌日への申し送りはjournal `next` で役割を混ぜずに再開できる。
- 破壊的操作、設定変更、記憶追加の確認規約を守る。
- プロジェクト候補確認、一般PJ作成、フル昇格、別repo開発PJ接続は、確認前の副作用を0件にする。
- すべてのユーザー向け出力は日本語。コマンド名・ファイル名・エラー名は原形を保つ。
