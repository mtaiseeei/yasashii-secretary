# Constraints

横断制約・禁止事項・安全方針。ここに書かれた条件は**回帰させてはならない不変条件**として扱う。
受け入れ済みスプリントで確認した不変条件は、状態ファイルではなくここに追記していく。

## 絶対ルール（CLAUDE.md 由来・最優先）

1. **`~/workspace/agentic-harness` は変更禁止**。
   - 同梱する「やさしいハーネス」は、必ずこのリポジトリ内に**複製してから**改変する。元リポジトリは読むだけ。
2. **外部データのローカル同期層を作らない**（my-vault の `10_sources` 型）。
   - 外部データ（メール・予定・ファイル・タスク）は公式リモートコネクタで**都度参照**する。同期コピー・キャッシュ層を作らない。GitHub Actions に依存しない。
3. **ターゲットは非エンジニア**。
   - ユーザー向け文言は「日常語＋具体例」。専門用語は必ず言い換えを併記する。
   - 報告は「**3行以内・専門用語は言い換え併記・次に何が起きるかを一言**」。

## 生成物（秘書ワークスペース）が守る規律（DESIGN.md「AGENTS.md に入れる規律」）

これらは、プラグインが生成する `secretary/AGENTS.md` に必ず盛り込む。

1. **スコープ**: 秘書は `secretary/` 配下だけ読み書きする。外は明示指示時のみ。資格情報は常時禁止。
2. **根拠ルール**: 要約・判断にはコネクタ由来の根拠（サービス名・リンク・日付）を明示する。原文にない事実を補完しない。矛盾は両方提示する。
3. **出力規約**: ファイル名 `YYYY-MM-DD_<title>.md`、frontmatter 必須（`createdAt` / `tags`）、1ファイル1トピック、見出しに固有名詞。
4. **記憶保護**（cc-company 継承）: 空内容で上書きしない。削除前に警告する。`MEMORY.md` 索引を常に最新に保つ。
5. **自動コミット**: 作業の節目ごとに日本語メッセージでローカル commit。**push はユーザーの明示指示時のみ**。
6. **報告の型**: 3行以内・専門用語は言い換え併記・次に何が起きるかを一言。

## プライバシー・安全

- **push しない**（自動コミットはローカルのみ）。プライバシー懸念を生む外部送信を既定にしない。
- コネクタ経由で取得した外部データの**本文をローカルファイルに保存しない**（根拠は「サービス名＋URL/ID＋日付」で引用）。
- 資格情報（トークン・パスワード・API キー）を秘書のワークスペースに書き込まない・コミットしない。
- 破壊的操作（ファイル削除・上書き・大量変更）は、実行前に非エンジニアにわかる言葉で確認する。
- **封じ込め（不変条件）**: 記憶ツール等の書き込み/削除は、対象の**実解決先**（symlink 解決後の正規化パス）が `secretary/` 境界の内側にある場合のみ許可する。境界外を指す symlink 越えの書き込み/削除は拒否する（`exit 3`）。`.`・空・親方向へ脱出する相対参照は**偽装成功させず非ゼロで拒否**する（`exit 0` を返さない）。境界文字列の前方一致だけに頼らない（接頭辞衝突を避ける）。

## 構成上の制約

- インストール導線は cc-company と同じ**3コマンド**（marketplace add → install → `/secretary`）を維持する。
- SKILL.md は**薄いルーター**にし、機能ごとに段階ロードする（cc-company の 936 行単一 SKILL.md 方式は踏襲しない）。
- 「秘書＋道具箱」メタファーを保つ。部署制・キーワード振り分け表・部署間 inbox 通知・`case-NNN` 必須生成・`patterns/` 自動統合は**導入しない**。
- 秘書の作業フォルダは**見えるフォルダ・英語名**（`secretary/` 配下。隠しフォルダにしない）。
- **配布されるファイル（雛形・スキル・ルール等）は `plugins/cc-secretary/` 配下に置く**。SKILL からの同梱ファイル参照は `${CLAUDE_PLUGIN_ROOT}` 相対に統一し、リポジトリ直下の相対パスに依存しない（実インストール時の参照断を防ぐ。詳細は `docs/spec/domain.md`）。
- ライセンスは **public / MIT**。必須クレジットは**単段**: 元作者 **Shin-sibainu/cc-company（MIT）** のみ。
  - 既存作法にならい、`marketplace.json` の `forkedFrom` フィールド・README・（あれば）`package.json` に `forkedFrom`（Shin-sibainu/cc-company）と MIT を残す。
  - 中間フォーク **inoshinichi/bootcamp-company** は**必須クレジットに含めない**（分析の参考にした事実を docs 側に残すのは可。配布物のクレジット要件からは外す）。
  - **回帰チェックはこのクレジット方針を明示的に検査する**（配布物のクレジットが Shin-sibainu/cc-company・MIT を含み、inoshinichi/bootcamp-company を必須クレジットとして掲げていないこと）。

## やさしいハーネス（Phase 3）の平易化3点（DESIGN.md）

同梱するハーネスフォークは、裏側の docs/spec・sprint 契約などの技術的文脈は維持しつつ、以下を平易化する。

1. **Planner のヒアリングを日常語化**（質問・選択肢に具体例を付す）。
2. **報告の型を固定**（全エージェントが「3行以内・言い換え併記・次の一言」ルールファイルを参照）。
3. **進行の見せ方**（「いま計画→道具→確認→結果のどこか」を毎回宣言。第1回座学の語彙と接続）。

## この spec 群に関する制約

- Planner は「何を作るか」に集中し、スタック・スキーマ・具体的コンポーネント構造を spec に先取りで固定しない。
- 製品はマークダウン資産（スキル・テンプレート・ルール）であり、実行アプリ・サーバー・独自 UI ではない。
