# 共通表現rule

このruleは `common-core` が所有し、両editionで共有します。対象editionのstyleを適用しても、
一般技術用語、説明順、wizard共通copyを分岐させません。

## 標準語彙

- 日常語をベースにしつつ、一般に通じる技術用語はそのまま使います。過度な平易化をせず、
  無理な言い換えや幼稚なメタファーを避けます。
- そのまま使う語: ディレクトリ、フォルダ、ファイル、インストール、コミット、プラグイン、
  コネクタ、TODO、カレンダー、メール、リンク、設定。
- OAuth、リポジトリ、frontmatter、MCP、トークン、環境変数等、本当に馴染みの薄い語だけ、
  初出時に簡潔な補足を1回添えます。毎回は繰り返しません。
- 住まい・擬人化等の比喩で技術概念を置き換えません。`secretary/` は「秘書ディレクトリ」または
  「秘書フォルダ」と呼びます。

## 説明と進行

- 最初に「何が起きているか」「何をすればよいか」を示し、背景説明は必要な分だけ後に置きます。
- ユーザー向けのすべての応答は、内容のまとまりが見えるMarkdownにします。短い1要点だけなら
  自然な1段落で十分です。複数の手順、選択肢、変更点、結果、原因、影響、次の行動は、
  空行で分けた段落、箇条書き、または番号付き手順で構造化します。
- 複数要素を改行なしの長い平文へ連結しません。一方で、1文ごとの箇条書き、不要な見出し、
  同じ内容の繰り返し、装飾目的の強調は避けます。情報構造に必要な分だけMarkdownを使います。
- この可読性はユーザーの好みとして質問せず、`preferences.md` の口調、専門用語、報告の詳しさでも
  無効にしません。edition固有の内容や説明順は保ち、読みやすさだけを共通の最低基準にします。
- 成功、部分失敗、エラーでは、完了したこと、未完了のこと、原因、影響、次の行動のうち
  該当する要素を分けて示します。検索結果や診断結果は複数件なら一覧にし、根拠を各項目に添えます。
- 手順が複数turnにまたがり入力・許可を待つときは、計画 → 道具 → 確認 → 結果の現在地を一言で示します。
  これは段階の呼び名であり、正式な技術用語を隠すものではありません。
- 同じturn内のRead、routing、read-only確認では途中メッセージを出さず、最終応答へまとめます。
- 入力や許可を待つ必要がある操作では、時間、確認、失敗しうる箇所を始める前に一言で示します。
- 質問は2〜3択を基本にし、抽象語だけでなく具体例を添えます。
- 英語エラーは「何が起きたか」「次にすること」の順で日本語にし、検索用の正式名称は残します。

## 内部recordとの境界

journal、index、commit message、設定値、machine-readable出力など、機械が読む1件1recordの形式は
この会話可読性の対象外です。内部形式を複数行へ変えず、ユーザーへ説明するときだけ上のMarkdown構造を使います。

## common by design

Chatwork／Google Chat wizardのheading、body、label、CTA、empty、loading、error、success copyとDOM、focus、
accessibilityはcommon by designです。`copy/yasashii.json` の4面へ含めず、edition別に変更しません。
