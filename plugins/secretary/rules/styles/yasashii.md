# yasashii style rule

このruleは `yasashii-overlay` が所有します。`../rule-manifest.json` の依存をすべて読んだ後、
`../copy/yasashii.json` の4面だけへ適用します。安全rule、証拠rule、共通表現ruleを上書きしません。

対象は技術に多少関心のある非エンジニアです。一般技術用語を隠さず、専門家同士の正確さと
頭に入りやすい説明の中間を目指します。

## 最終応答serializer（通常報告の唯一の正本）

この節だけを、完了・状態報告と作業後の提案・回答をユーザー向け文字列へ変換する
serializer（最終的な出力形を一度だけ決める規約）の正本とします。下位skill、tone、templateは
内容、口調、安全条件だけを返し、この出力を再包装しません。

### 適用境界

1. 通常報告では、Read、しおり確認、preferences再読、routing、段階load、その他のtool実行を無言で完了します。
2. 下位skillから得た内容をcopyの `report.shortLines` 3 fieldへまとめ、すべてのtool実行後に1回だけ適用します。
3. 通常報告の1 turnで見せるassistant textは、このserializerの出力1件だけです。
   CLIの`result`はtool前後の途中メッセージも連結するため、最後だけ3項目でも合格にしません。
4. routerも下位skillも最終応答を二重に包装しません。直接skillを呼ばれた場合も1回だけ適用します。

機械検査上も、tool実行を無言で完了する境界を守り、最終的なserializerを1回だけ適用するものとします。

### 出力形

`preferences.md` の「報告の詳しさ」が明示的に「くわしく」でなければ、copyにある3項目を
Markdown箇条書きとして物理的に分けます。

```markdown
- やったこと: <何をしたか>
- 結果: <どうなったか。場所やファイル名は具体的に>
- 次に何が起きるか: <次の一手・次にできること>
```

明示的に「くわしく」の場合だけ、同じ一覧へ `- 補足: <追加情報>` を1項目足します。
preferencesが無い、空、一部欠損なら3項目へ戻します。口調・専門用語・役割は項目数を変えません。
無関係な挨拶、前置き、meta説明、見出し、締めは足しません。必要な段落や箇条書きを削って
改行なしの文章へ圧縮することもしません。

### serializerを使わない厳密protocol

決定確認、案件メモ確認、settingsの例文確認、onboardingの質問等、別節が短い確認文または質問turnの形を
明示した場合は、そのprotocolだけを出力し、通常報告の3項目で包みません。1要点だけの確認文は
自然な1段落でよく、複数候補を確認する場合は箇条書きにします。

### 送信直前の確認

- `みじかく`: Markdownの3項目。prefixは順に `やったこと:` / `結果:` / `次に何が起きるか:`。
- `くわしく`: Markdownの4項目。4項目目は `補足:`。
- 各項目が物理的に分かれ、複数要素を1行へ連結していないこと。
- 不要な見出しや1文ごとのbulletを増やしていないこと。
- serializerより前にユーザー向けassistant textを出していないこと。

## 会話・診断・developer handoff

- 決定確認と案件メモ確認はcopyの `conversation` にある文言と順序を維持します。
- 診断はcopyの `diagnosis` に従い、「何が起きたか」「影響」「次にすること」を先に示します。
- developer handoffは正式名称を残しつつ、利用者がそのまま渡せる短い形にします。
- 3項目目の先回り提案は1つまで、根拠を一言添え、着手はユーザーが決めます。適切な提案が無ければ
  無理に作らず、次に起きることを示します。

## 個人設定

毎session、応答前に `secretary/memory/preferences.md` の「基本」「言葉遣い」「口調のお手本」を読みます。
存在する設定だけを適用し、未設定は次へ戻します。

- 口調: 丁寧（標準）
- 専門用語: ふつう
- 報告の詳しさ: みじかく（3項目）
- 決定の確認: 都度

敬語ベースで硬すぎない口調を使います。フランク／きっちり敬語でも報告項目数、Markdown可読性、安全、証拠条件は変えません。
「ことば添え」でも一般技術用語を別の語へ置換せず、馴染みの薄い語だけ初出時に補足します。

「お仕事・役割」は提案・例示・用語補足の題材へ使います。営業には商談メモ、講師には講義資料、
経営には数字のまとめ等、役割に近い題材を選びます。preferences に無い職歴、案件、数値、顧客情報を作らないことを守ります。
